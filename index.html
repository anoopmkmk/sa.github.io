The Sweet Assist API provides programmatic access to read transaction details including CDA form, Detail page, Documents etc.

To use this API, you need an API key. Please contact us at dev@sweetassist.com to get your own API key.

List transaction data using API

Sweet Assist API provides a seamless way to integrate transaction data into your website, allowing businesses to efficiently manage and display financial records. With its secure and user-friendly endpoints, developers can retrieve transaction histories, filter records by date, amount, or status, and display real-time updates. By making authenticated API calls, businesses can ensure secure data transmission while enhancing transparency for users. Whether you're managing e-commerce payments, tracking invoices, or monitoring financial activities, Sweet Assist API simplifies data retrieval and integration, making it a powerful tool for transaction management on modern web platforms.

Authentication Scheme

The Sweet Assist API Authentication Schema ensures secure and reliable access to transaction data by implementing robust authentication mechanisms. API key-based authentication, developers can authenticate requests while preventing unauthorized access. Each request requires a valid Bearer Token or API key in the header, ensuring that only authorized users can retrieve or modify transaction records. With built-in token expiration and refresh capabilities, the schema enhances security while maintaining seamless user experiences. By enforcing encrypted communication over HTTPS and supporting role-based access controls, Sweet Assist API authentication provides a scalable and secure foundation for integrating financial data into modern web applications.

Request/Response scheme

The Sweet Assist API Request/Response Scheme provides a structured and secure way to interact with transaction data. By utilizing a RESTful architecture, requests are authenticated using API keys, ensuring only authorized access to sensitive financial records. A standard API request includes an authentication token in the header, followed by a well-defined endpoint to fetch transaction details. The API responds with a structured JSON payload, containing transaction IDs, amounts, statuses, timestamps, and customer details. In case of errors, the API returns clear HTTP status codes and messages, enabling efficient error handling. This robust scheme ensures seamless integration, real-time data retrieval, and secure financial operations for modern web applications.

Read access (Currently we are providing read access only)

The Sweet Assist API currently offers read-only access, allowing users to securely retrieve transaction data without making modifications. This ensures data integrity while providing real-time insights into financial records. Through authenticated API requests, developers can fetch transaction histories, filter data by date, status, or amount, and access detailed customer information. The read access model enhances security by preventing unauthorized changes while enabling seamless integration with dashboards, reporting tools, and financial analytics platforms. By focusing on data retrieval rather than modifications, Sweet Assist API ensures a reliable and transparent way to access essential transaction data for business intelligence and monitoring.

AUTHENTICATION

The Sweet Assist API provides programmatic access to read transaction details including CDA form, Detail page, Documents etc.

Client Registration

To register your application to integrate with this API, please contact dev@sweetassist.com. Upon registration, we will issue you a client id and client secret which are prerequisites in order to use the API.

Generate Authorization Code

To begin, direct the user to the authorization URL in their browser.

Request Example:

GET https://api-staging.sweetassist.com/v1/public/oauth/authorize

?client_id=YOUR_CLIENT_ID

&redirect_uri=YOUR_REDIRECT_URI

&response_type=code

Parameters:

client_id - Your registered API client ID.

redirect_uri – The URL where the user will be redirected after authorization.

response_type – Specifies that an authorization code should be returned.

Response Example:

{

"status": true,

"code": "wFzb1bhdBvEbBQEvmvyd",

"redirect_uri": "https://dev.sweetassist.com/admin/broke",

"message": "Success"

}

User Grants Permission:

The user will see a consent screen and approve access.If successful, the API will redirect the user to your provided redirect_uri with the authorization code in the URL.

Example Redirected URL: https://yourwebsite.com/callback?code=AUTHORIZATION_CODE

Obtaining Access token

To securely interact with the Sweet Assist API, users must first obtain an access token. The access token is required to authenticate API requests and is included in the request headers for secure data retrieval.

1.Token Request

Send a POST request to the authentication endpoint with your client credentials.

Request Example

To obtain an access token, send a POST request to the authentication endpoint with your client credentials.

POST https://api-staging.sweetassist.com/v1/public/auth/token

Content-Type: application/json

{

"code": "code",

"redirect_uri": "your_redirect_uri",

"grant_type": "authorization_code"

"state": "state"

}

Response Example

Content-Type: application/json

{

"status": true,

"access_token": "eyJ0eXAiOiJKV1QiLCJhbGciOiJSUzI1NiJ9....",

"refresh_token": "3e4348b5b0e5fad60e49f603a4a27....",

"token_type": "Bearer",

"expires_at": "2026-02-07T09:04:23.000000Z",

"message": "Authentication successful"

}

Obtaining Refresh token

Once the access token expires, use the refresh token to get a new access token.

POST https://api-staging.sweetassist.com/v1/public/auth/token

1.Token Request

Send a POST request to the authentication endpoint with your client credentials.

Request Example

To obtain an refresh token, send a POST request to the authentication endpoint with your client credentials.

POST https://api-staging.sweetassist.com/v1/public/auth/token

Content-Type: application/json

{

"grant_type": "refresh_token",

"client_id": "YOUR_CLIENT_ID",

"client_secret": "YOUR_CLIENT_SECRET",

"refresh_token": "YOUR_REFRESH_TOKEN"

}

Response Example

Content-Type: application/json

{

"access_token": "NEW_ACCESS_TOKEN_VALUE",

"expires_in": 3600,

"token_type": "Bearer",

"refresh_token": "NEW_REFRESH_TOKEN_VALUE"

}

Refreshing Access token after expiration

When an access token expires, you can use the refresh token to get a new one without requiring the user to log in again. This process ensures seamless authentication and uninterrupted API access.

1.Best Practices

  • Store Tokens Securely: Keep the refresh token safe as it allows re-authentication.
  • Auto Refresh Before Expiration: Use the expires_in field to refresh proactively.
  • Handle Expired Refresh Tokens Gracefully: If the refresh token is also expired, prompt the user to log in again.

Content Type

The Content-Type header in an API request specifies the format of the data being sent to the server. The Sweet Assist API primarily supports application/json for sending and receiving data securely and efficiently.

1.What is Content-Type?

  • The Content-Type header informs the API about the format of the data included in the request body. It ensures the server can correctly interpret and process the request. For the Sweet Assist API, the supported content type is:.
    • Content-Type: application/json
  • This means the API expects requests to be formatted in JSON (JavaScript Object Notation).

2.When to Use Content-Type: application/json

You should include the Content-Type: application/json header when making API requests that contain a request body, such as:

  • Authentication & Token Requests (e.g., refreshing an access token).
  • Creating or Updating Data (e.g., adding a new transaction).
  • Filtering or Querying Data (e.g., searching for transactions based on criteria).

  • Always specify Content-Type: application/json when sending JSON data.
  • Validate JSON format before sending the request to avoid syntax errors.
  • Use HTTPS to ensure secure transmission of sensitive data
  • Check API documentation to confirm if additional headers are required

Authorization header

The Authorization header is a crucial part of API security, ensuring that only authenticated users can access protected endpoints. In the Sweet Assist API, authentication is done using Bearer Tokens, which must be included in every request requiring authentication.

1.Format of the Authorization Header

  • When making API requests, the Authorization header should follow this format:

    • Authorization: Bearer YOUR_ACCESS_TOKEN

  • Bearer: This specifies that the token is an OAuth 2.0 Bearer Token.
  • YOUR_ACCESS_TOKEN: Replace this with the valid access token obtained from the API.

2.Common Authorization Errors

4.Best Practices for Using Authorization Header

  • Always include Authorization: Bearer YOUR_ACCESS_TOKEN in authenticated requests.
  • Store access tokens securely (e.g., environment variables, secure storage).
  • Refresh tokens proactively to avoid interruptions.
  • Use HTTPS to secure token transmission.

FETCH DATA USING API

The Sweet Assist API provides programmatic access to read transaction details including CDA form, Detail page, Documents etc.

List all transactions

To retrieve all transactions from the Sweet Assist API, you need to make an authenticated GET request to the transactions endpoint.

Request Example:

GET https://api-staging.sweetassist.com/v1/public/transaction

?response_type=data

&batch_size=10

&batch_number=1

Parameters:

batch_size - Number of transactions in a single page.

batch_number – Which page data wants to load.

response_type – Specifies that an authorization code should be returned.

"Authorization: Bearer YOUR_ACCESS_TOKEN"

"Content-Type: application/json"

Get single transaction details.

To retrieve details of a specific transaction from the Sweet Assist API, you need to make an authenticated GET request to the transaction details endpoint using the transaction ID.

Request Example:

GET https://api-staging.sweetassist.com/v1/public/transaction/detail

?response_type=data

&transaction_id={{transaction_id}}

Parameters:

response_type – Specifies that an authorization code should be returned.

transaction_id=The Id of the single transaction.

"Authorization: Bearer YOUR_ACCESS_TOKEN"

"Content-Type: application/json"
